# zf_info navigation TOCs

Ordered **navigation manifests** (RWPM subset) for the `zf_info` collections —
step 1 of the page-nav project (adding "back / forward / next section /
back-to-toc" links to the static reference pages).

Generated by [`build-toc.py`](build-toc.py) (in this `toctools/` dir), writing
the JSON into `src/zf_info/toc/` so Apache serves it at **`/zf_info/toc/*.json`**
(the client-side nav can `fetch` it):

```sh
python3 toctools/build-toc.py            # all collections
python3 toctools/build-toc.py monitor    # one collection
```

(`toctools/` is a scratch home for the parser for now; it will be removed later.)

Each `src/zf_info/toc/<collection>.json` is a **subset of the Readium Web
Publication Manifest** ([RWPM](https://readium.org/webpub-manifest/)) — enough to
name fields by an established standard without chasing full conformance (no
`@context` / media types):

- `metadata` — `{title, identifier}` (and `description` on the hand-authored
  `loose`/`misc` files).
- `links` — the collection's source contents page as `[{rel: "contents", href}]`
  (drives back-to-toc). Omitted for `loose`/`misc` (no single contents page).
- `readingOrder` — the ordered, de-duplicated linear page sequence:
  `[{href, title}]`. Drives prev/next (index ±1). *(RWPM's reading order; was
  `spine`.)*
- `toc` — the same pages grouped by heading, in order:
  `[{title, children: [{href, title}]}]`. *(RWPM's table of contents; was
  `sections`/`pages`.)*
- `coverage` — non-standard extension: `readingOrderPages` vs
  `collectionHtmlPages` (pages on disk).

The parser walks a collection's contents page in document order, reading both
`<a href>` and `<area href>` (anatomy uses image maps). For hierarchical
collections it follows per-section sub-index pages (`follow` regex) and inlines
their pages.

## Collections

| Collection | Seed contents page | readingOrder | Notes |
|---|---|---|---|
| `zfbook`  | `zfbook/cont.html`          | 80 | The Zebrafish Book chapter spine. Its other pages are the staging series (separate collection below). Chapters 6 & 10 are single pages. |
| `staging` | `zfbook/stages/stages.html` | 63 | Developmental Staging Series. Gaps: `table1/2.html`, `larval*.html` (not linked from the seed); `index.html`/`stages.html` are TOC pages, excluded. |
| `monitor` | `monitor/mon.html`          | 93 | The ZFIN Monitor. Recurses into per-volume `cont.html` indexes. All content pages covered; the 6 non-spine pages are `mon.html` + the volume `cont.html` nav nodes. |
| `anatomy` | `anatomy.html`              | 79 | Developmental anatomy atlas (24/48/72/120-hr stages). Recurses into the hour pages' image maps. Order is **image-map (spatial)**, not numeric — a future refinement could sort by stage number. The ~16 `anatomy/dict/` pages are a separate dictionary sub-collection (not linked from the hour maps) and are not included. |

## Not covered (loose collections)

These have **no single ordered contents page**, so there is no meaningful spine
to extract — they are standalone pages, not a sequenced book:

- `catch/` (10), `news/` (8), `sequence/` (2), `dbase/PAPERS/` (5)
- misc single pages: `glossary`, `nomen`, `syntax_help`, `stars`, `zfin_stats`,
  `defs`, `blast_info`, `author_guidelines`, `author_guide_checklist`,
  `movies/`, `2006_tutorial/`, `zfprbs`

For these, page-nav (if wanted) would use a hand-authored order or just
back-to-collection-landing, not an auto-extracted spine.

## Status

This is a build-time artifact checked in for review. When the nav project starts,
`build-toc.py` can be wired into the site build (`npm run build`) so the manifests
regenerate from the content, and the nav links can be injected client-side (like
the chrome header/footer) or baked into the layout.
